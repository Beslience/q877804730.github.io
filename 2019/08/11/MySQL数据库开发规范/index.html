MySQL数据库开发规范

发表于 2019-08-11 | 分类于 Notes

MySQL数据库开发规范

数据库命名规范

  • 数据库对象名称小写并且使用下划线分割

  • 临时库表以tmp_为前缀以日期为后缀，备份表以bak_为前缀以日期为后缀

  • 数据库库表名称禁止使用MySQL保留的关键字，名称见名知意，不超过32个字符

数据库表基本设计规范

  • 所有表必须使用Innodb存储引擎（支持事物控制、行级锁、有更好的恢复性，高并发下性能更好）

    • 除无法满足列存储、存储空间数据等
    • MySQL5.5之前默认使用MyIsam，5.6后默认使用Inoodb

  • 数据库和表使用统一的uft8mb4（避免由于字符集转换产生的乱码，转换会造成索引失效）

  • 单表数据控制在500W行以内（过大会造成修改表结构、备份、恢复缓慢）

  • 表中做到冷热数据分离，减少表的宽度

    • 表越宽，将表数据装载进内存需要消耗更多的内存、更多的磁盘IO

  • 禁止在表中存储图片、文件大的二进制数据

  • 禁止在线上对生产数据库进行压力测试

  • 禁止从开发、测试环境连接生产环境数据库

数据库表字段设计规范

  • 优先选择符合存储的最小数据类型

    • 字段越大，建立索引需要使用的空间越大，这样一页存储中的节点数据更少，需要IO读取更多次才能找到数据
    • 举例：对于非负数型的数据，要优先使用无符号整型进行存储
      • SIGNED INT -2147483648~2147483647
      • UNSIGNED INT 0~4294967295

  • 尽可能将所有列设置为NOT NULL

    • 索引NULL列需要使用额外的空间保存，要占用更多的空间
    • 进行比较和计算需要NULL进行特别的处理

  • 使用TIMESTAMP（4个字节）和DATETIME（8个字节）存储时间

    • TIMESTAMP 存储的时间范围 1970-01-01 00:00:01 ~ 2038-01-19-03:14:07
    • TIMESTAMP 占用4字节和INT相同，但比INT可读性高
    • 超过TIMESTAMP取值范围使用DATETIME类型进行存储
    • 不要使用字符串存储日期类型数据
      • 无法使用日期函数进行计算和比较
      • 使用字符串需要占用更多的空间

  • 同财务相关的金额类型数据需要demical类型

    • demical为精准浮点数，在计算时不会丢失精度

索引设计规范

  • 限制每张表上索引的数量，单张表不超过5个
    • MySQL优化器在查询时，会对每一个索引进行评估，生成最好的执行计划，如何有很多个索引，会增加MySQL优化器生成执行计划的时间，降低查询性能
  • 常见列索引建议
    • 出现在SELECT、UPDATE、DELETE从句中WHERE从句中的列
    • 出现在ORDER BY、GROUP BY、DISTANCT中的列
    • 多表join的关联列
  • 联合索引顺序
    • 区分度最高的放在联合索引左侧
    • 字段长度小的列放在联合索引左侧
    • 使用频繁的列放在联合索引左侧
  • 优先考虑使用覆盖索引
    • 覆盖索引：包含where、select、order by、group by字段的索引

SQL开发规范

  • 使用预编译语句进行数据库操作

    • 预编译语句可以重复这些计划，减少SQL编译使用的时间
    • 解决动态SQL带来的SQL注入问题

  • 避免数据的类型的隐式转换

    • 隐式转换会导致索引失效：

      select id from student where id = ‘111’;

  • 充分利用表上已经存在的索引

    • 避免使用双’%’的查询条件：name like ‘%123%’（无前置%，只有后置%，可以使用列上的索引）
    • 一个SQL只能利用联合索引的一列进行范围查询
      • a,b,c列的联合索引，在查询条件中使用a列范围查询，在b,c列上的索引不会使用到
      • 在定义联合索引，如果需要使用a列进行范围查询，就将a列放在右侧
    • 使用left join或not exist代替not in，因为not in不会使用索引

  • 避免使用子查询

    • 子查询时会生成临时表，不会为临时表（内存/磁盘）生成索引，消耗更多的CPU和IO资源，产生慢查询

  • 避免join关联太多的表，控制在5个以内

    • MySQL存在关联缓存，同一个SQL关联一个表会分配一个关联缓存，当关联表越多时，会占用更多的内存，影响服务器性能

数据库操作规范

  • 超过100万行的批量操作（INSERT、UPDATE、DELETE）需要分批次进行

    • 主从环境时，会造成严重的操作延迟，大批量操作需要执行长时间，只有在主库执行完才会同步至从库
    • Binlog日志为row时会产生大量的日志，日志会记录每一行数据的修改
    • 避免产生大事物操作：大批量修改数据，会在同一个事物中执行，会导致大批数据被锁定，导致大量的阻塞
    • 长时间的阻塞会占满其他连接，导致其他应用无法使用数据库

  • 对于大表数据使用pt-online-schema-change修改表结构

    • pt-online-schema-change会建立一个原表结构相同的新表，然后对新表的结构进行修改，再复制旧表中的数据到新表，复制完成后将新表名字修改为原表名字，将旧表删除

  • 禁止一个应用使用的权限为super权限

    • 当连接达到最大时，需要剩一个连接留给DBA的super权限

  • 一个数据库账号只能操作一个数据库，不进行有跨库drop的权限
